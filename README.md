**원인**
같은 슬롯에 요청이 몰리면 DB row lock을 두고 경합이 발생해, 대기가 쌓이며 처리량이 떨어졌다. Redis 선차감으로 옮기자 이번엔 GET-DECR 사이 끼어듦으로 과예약이 발생했고, Lua Script로 묶자 단일 스레드 특성상 성능 저하가 생겼다.

**해결**
GET 없이 DECR 한 번으로 차감하고, 음수면 INCR로 롤백하는 단일 카운터 방식을 택했다.

## 대기열 KEYS 병목 제거

**문제**
대기열 스케줄러가 활성 팝업을 찾을 때 `redisTemplate.keys()`로 전체 키를 탐색하고 있었다. KEYS는 프로덕션에서 지양되는 명령이다.

**원인**
KEYS는 전체 키를 순회하는 O(N) 명령인데, Redis는 단일 스레드라 실행 중 다른 모든 요청이 블로킹된다. 키가 늘면 정원 차감 같은 핵심 명령까지 멈출 수 있다.

**해결**
활성 팝업 ID만 담는 Set 인덱스로 전체 탐색을 없앴다. 빈 팝업을 즉시 삭제하면 "확인 → 제거" 사이 새 대기자가 누락될 수 있어, 조회 시점에 지우는 lazy deletion을 택했다. 잘못 지워져도 다음 대기자가 들어올 때 자동 복원된다.

**고찰**
lazy deletion에 미세한 경쟁 조건은 남지만, 발생 조건이 좁고 자연 복원돼 실익이 적다. 완전한 원자화(Lua)는 구현 공수 대비 발생 가능성이 낮아 현재는 도입하지 않았다.

## 단일 장애점(Redis) 대비 — 장애 격리와 복구

**문제**
Redis에 장애가 생기면 서버가 응답을 기다리며 타임아웃마다 스레드를 낭비하고 요청이 쌓일 수 있다고 예상했다.

**원인**
Redis 호출은 응답까지 스레드가 대기하는 구조라, 장애 시 스레드가 타임아웃까지 묶이고 결국 스레드 풀이 고갈돼 요청까지 막힐 수 있다.

**해결**
서킷브레이커를 도입해 장애를 감지하면 요청을 빠르게 차단했다. 무의미한 대기와 스레드 점유를 막고, 장애가 전체로 전파되는 것을 방지했다.

<!-- 이미지: Redis 장애 → 서킷브레이커 감지 → 요청 차단 흐름 다이어그램 -->
<!-- ![서킷브레이커 장애 격리](images/popspot-circuit-breaker.png) -->

**고찰**
현재는 실제 운영환경이 아니라 DB 기준 재구축을 수동 복구로 두었지만, 실제 운영이라면 복구 중 플래그로 신규 요청을 막고 Redis Sentinel·Cluster 등 고가용성을 구성하는 지점도 고려해야 한다.

## 운영 종료일 기준 TTL로 카운터 키 생명주기 관리

**문제**
재고 카운터 키를 언제까지 유지할지 정해야 했다. 너무 짧게 잡으면 이후 취소·환불 처리 중에 키가 사라져 더 큰 문제가 될 수 있었다.

**원인**
슬롯 종료 시각을 기준으로 잡으면, 운영 기간 내내 발생하는 취소·환불 시점에 이미 키가 만료돼 재고를 되돌릴 수 없게 된다.

**해결**
TTL을 슬롯 종료가 아닌 운영 종료일 기준으로 설정해 취소·환불 여유를 확보했다. 동시 만료 시 CPU 부하도 검토했으나, 예상 규모(약 7,000개 키)에선 문제없다고 판단했다.

**고찰**
지금 규모에선 충분하나, 팝업·슬롯 수가 크게 늘면 동시 만료 부하가 커질 수 있다. 그 경우 만료 시점을 분산하거나 Redis를 용도별로 분리하는 방식을 고려할 수 있다.

## 트랜잭션 커밋 이후로 S3 이미지 처리를 미뤄 정합성 보장

**문제**
팝업스토어 등록·수정·삭제 시 S3 이미지를 이동하거나 삭제해야 한다. 이를 트랜잭션 안에서 바로 처리하면, 이후 DB 작업이 실패해 롤백되어도 이미 옮기거나 지운 파일은 되돌릴 수 없다.

**원인**
DB는 롤백으로 복구되지만 S3 파일 조작은 외부 시스템이라 롤백 대상이 아니다. 트랜잭션 중간에 파일을 건드리면 DB 상태와 실제 파일이 어긋날 수 있다.

**해결**
이미지 이동·삭제를 `afterCommit` 시점으로 미뤄 커밋된 뒤에만 처리했다. 롤백되면 파일이 그대로 유지돼 어긋나지 않는다. 트랜잭션 없이 호출되는 경우를 대비한 fallback(즉시 실행)도 두었다.

**고찰**
현재는 move 실패 시 로그만 남는다. 재시도보다 실패를 감지할 수단이 우선이라 봤고, 이미지 기능의 비중이 커지면 재시도·Outbox 같은 복구 강화를 단계적으로 도입할 수 있다.

## 재고 동시성 제어 — 상황별 락 전략

**문제**
예약 시스템에서 재고 차감과 반환은 동시성 부하가 다르다. 오픈런 재고 차감은 같은 자리에 수천~수만 명이 몰려 충돌이 극심하지만, 재고 반환(취소·수정)은 본인 예약 대상이라 충돌이 드물다. 하나의 락 전략을 일괄 적용하면 한쪽에서는 반드시 비효율이 발생하는 상황이었다.

**원인**
동시성 제어에서 충돌은 여러 요청이 같은 자원(정원)을 동시에 변경하려 경합하는 상황을 말한다. 두 락은 이 충돌의 빈도에 따라 유불리가 갈린다. 낙관적 락은 충돌이 드물 때 락을 미리 걸지 않아 비용이 거의 없지만, 충돌이 잦으면 재시도가 누적돼 성능이 떨어진다. 분산 락은 충돌을 사전에 차단해 충돌이 잦은 상황에 강하다. 따라서 충돌 빈도를 기준으로 락 전략을 나눴다.

**해결**

**차감(충돌 잦음) → 분산 락.** 오픈 시 다수가 같은 자리에 동시에 몰려 충돌이 빈번하므로, 재시도 폭증을 막는 분산 락을 적용했다. k6로 200 VU가 재고 1개 슬롯에 10,000건 동시 요청하는 시나리오로 검증했고, 정합성은 두 방식 모두 CONFIRMED 1건으로 완벽했으나 성능은 분산 락이 1,390 req/s로 낙관적 락(321 req/s) 대비 4.3배 우세해 채택했다.

**반환(충돌 드묾) → 낙관적 락(@Version).** 각자 자신의 예약을 취소하는 흐름이라 같은 자리에 충돌이 몰릴 확률이 낮으므로, 불필요한 락 비용이 없는 낙관적 락으로 충분했다. 50개 동시 취소 요청에도 정확히 1건만 성공하고 나머지는 차단되며 재고가 정확히 복구됨을 확인해, 분산 락 없이 낙관적 락만으로 충분함을 입증했다.

<!-- 이미지: 차감/반환 락 전략 비교 표 또는 성능 그래프 (분산락 1390 vs 낙관락 321 req/s) -->
<!-- ![락 전략 검증 결과](images/mozu-lock-perf.png) -->

**고찰**
같은 시스템 안에서도 동시성 충돌 빈도에 따라 도구를 다르게 가져가는 것이 중요하다고 판단했다. 반환처럼 충돌이 드문 곳에 분산 락을 쓰면 안 나는 충돌을 막느라 비용만 낭비되고, 차감처럼 충돌이 잦은 곳에 낙관적 락을 쓰면 재시도 누적으로 성능이 무너진다. 상황의 충돌 빈도에 맞는 최소한의 도구를 고르는 것이 핵심이었다.

## 취소 재고를 랜덤 반환해 즉시 선점 완화 (CANCEL_PENDING)

**문제**
취소 재고를 즉시 반환하면 반환 시점이 예측 가능해져, 매크로가 정상 사용자보다 유리하게 자리를 선점할 수 있다.

**해결**
어뷰징이 의심되는 취소는 즉시 반환하지 않고 CANCEL_PENDING 상태로 두어, 동적 스케줄러가 랜덤한 시간에 반환하도록 했다. 반환 시점을 예측 불가능하게 만들어 매크로의 즉시 선점을 어렵게 했다. 반환 수단으로는 Redis TTL을 고려했으나, TTL은 만료 시점에 콜백을 보장하지 않아 keyspace notification을 별도로 설정해야 하고 그마저 알림 유실 가능성이 있다. 이 경우 재고가 영영 반환되지 않을 위험이 있어, 반환 시점을 DB에 저장하고 서버 재시작 시에도 복구 가능한 DB 기반 동적 스케줄러로 구현했다.

## DB 인덱스로 조회 성능 개선

**문제**
사용자별 예약 조회(`GET /my/reservations`)가 데이터가 쌓일수록 느려졌다. 예약 100만 건 기준으로 p95 응답시간이 3.83초까지 치솟아, 조회 기능이 사실상 사용 불가능한 수준이었다.

**원인**
`WHERE user_id = ?` 조회가 인덱스 없이 실행되어 테이블 전체를 훑는 Full Scan이 발생했다. 100만 건을 매 요청마다 순회하니 응답이 느려지고, CPU 사용량도 99.6%까지 올라 DB에 큰 부하가 걸렸다.

**해결**
조회 조건인 `user_id` 컬럼에 인덱스를 추가해 Full Scan을 인덱스 탐색으로 전환했다. 인덱스는 조회를 빠르게 하는 대신 쓰기 시 갱신 비용이 있어, 조회가 빈번한 `user_id`에 선별적으로 적용했다.

**평가** — 100만 건 데이터, 100 VU 부하로 인덱스 적용 전후를 k6와 Grafana로 검증했다.
- **p95 응답시간:** 3.83초 → 9.24ms (목표 50ms 이하 달성)
- **CPU 사용량:** 99.6% → 거의 0에 수렴
- Full Scan이 사라지며 DB 부하가 근본적으로 제거됨

<!-- 이미지: 인덱스 전후 p95 / CPU 비교 그래프 (k6 + Grafana) -->
<!-- ![인덱스 성능 비교](images/mozu-index-perf.png) -->

**고찰**
인덱스 하나로 조회 병목을 해소했지만, 취소 스케줄러 쿼리(`findAllByStatusAndReleaseAtBefore`)에서도 Full Scan을 발견해 `status`·`release_at` 복합 인덱스가 추가로 필요함을 확인했다. 단순히 느린 쿼리에 인덱스를 거는 것을 넘어, 조회 패턴 전체를 점검해 인덱스를 설계하는 접근이 필요하다고 판단했다.

## 스케줄러 @Transactional 자기 호출로 인한 트랜잭션 미적용 해결

**문제**
재고를 반환하는 스케줄러에서 재고 반환과 예약 취소를 한 트랜잭션으로 처리해야 했는데, 실제로는 트랜잭션이 걸리지 않아 부분 실행 시 롤백되지 않는 위험이 있었다. 재고는 반환됐는데 예약 취소가 실패하면 데이터가 어긋난다.

**원인**
스케줄러가 같은 클래스 안의 `@Transactional` 메서드를 직접 호출하고 있었다. Spring의 `@Transactional`은 프록시를 통해 동작하는데, 같은 클래스 내부 호출(자기 호출)은 프록시를 거치지 않아 트랜잭션이 적용되지 않는다.

**해결**
트랜잭션이 필요한 로직을 별도 서비스 클래스로 분리했다. 다른 빈으로 분리되면 호출 시 프록시를 거치게 되어 `@Transactional`이 정상 동작한다. 이로써 재고 반환과 예약 취소가 하나의 트랜잭션으로 묶여, 중간 실패 시 함께 롤백되도록 원자성을 보장했다.

## AI 코드 리뷰 자동화로 리뷰 품질·속도 개선

**문제**
PR마다 사람이 수동으로 리뷰하다 보니 리뷰 누락이 생기고, 팀 컨벤션 위반을 매번 사람이 체크하기 어려웠다.

**해결**
GitHub Actions에 Gemini 기반 AI 코드 리뷰를 도입했다. 시니어 백엔드 개발자 페르소나를 부여하고, 팀 컨벤션(DTO/Entity data class 금지, `!!` 금지, Optional 남용 금지 등)을 자동 검증하도록 커스텀했다. 리뷰 결과는 심각도(필수/권장/참고)로 분류해 우선순위를 명확히 했다. 또한 CODEOWNERS와 랜덤 리뷰어 지정, Slack 알림을 연동해 리뷰 프로세스를 자동화했다.

**성과**
AI 리뷰가 DB 비밀번호 하드코딩, JWT 시크릿 키 노출 같은 보안 이슈와 타임존 버그, 재고 중복 반환 등을 실제로 잡아냈다.
